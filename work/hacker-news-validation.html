Case Study / Data Validation

Hacker News
Data Validation

BigQuery SQL Looker Studio

Context

Hacker News is a social news website focusing on computer science and entrepreneurship, run by the investment fund and startup incubator Y Combinator.

The Hacker News engineering team pushed a major backend update on October 1, 2024. The task was to validate that data capture looked normal.

Framework

Stage 01

Table Structure & Integrity

Schema review, primary key validation, duplicate check, and partitioning assessment.

Stage 02

High Level Event Count

Volume by event type and date against a 12-month baseline to identify anomalies.

Stage 03

Event Specific Parameters

Null rate tracking on key parameters for the two highest-priority events: story and comment.

Stage 01 / Table Structure & Integrity

Before touching any event data, I needed to understand how the table was organized and whether it could be trusted as a reliable foundation for analysis.

I examined the schema and scanned the table preview. All 14 columns are nullable except type and id, which are populated on every row. The type column is the event name — the primary dimension for any validation work. The id column is documented as the unique identifier, but there is no designated primary key enforcing that constraint.

Two queries confirmed the table's current integrity: one verified that no rows had null type or id values; the other confirmed no duplicate IDs exist at this time. Both returned clean. The absence of a primary key constraint is still a structural risk — nothing prevents duplicates from being introduced — and worth flagging for future joins against other tables.

A third finding: no partitioning column exists on the table. Working within a 1TB free tier quota, this meant full table scans on any unfiltered query. Structuring queries to filter on timestamp early was important for staying under quota.

Null check on type and id 
SELECT type, id
FROM `bigquery-public-data.hacker_news.full`
WHERE type IS NULL
OR id IS NULL;
Duplicate ID check 
SELECT
    id,
    COUNT(*) AS id_count
FROM `bigquery-public-data.hacker_news.full`
GROUP BY 1
HAVING COUNT(*) > 1
ORDER BY 1 DESC;

Stage 02 / High Level Event Count

With the table structure understood, the next step was validating whether event volumes looked reasonable — both in absolute terms and relative to historical patterns.

I pulled a count of events by type and date across the trailing 12 months, converting the UTC timestamp to EST. The most common event types are comment and story, which drove the prioritization for Stage 03. job is third but substantially lower in volume.

October 2024 hit a 12-month high in story events, with an acute spike on October 29. This kind of anomaly requires escalation before it can be interpreted. I would verify with the marketing and product teams whether there was a campaign, a viral story, or any other business reason for increased traffic. If UTM data exists, that's the first place to look. If no external explanation surfaces, it warrants deeper investigation into potential data capture issues.

One additional table-wide finding: approximately 27,000 rows are labeled as story events but carry no timestamp. This means story event counts are potentially undercounted by that amount, and the issue could be concentrated in a single period or spread across the table's history back to 2006.

Screenshot: Story Events — Last 12 Months (By Date)
Event count by type and date 
SELECT
    FORMAT_DATE('%Y-%m-%d', DATE(timestamp, "America/New_York")) AS event_date,
    type,
    COUNT(*) AS events
FROM `bigquery-public-data.hacker_news.full`
GROUP BY 1, 2
ORDER BY 1, 2 DESC;

Stage 03 / Event Specific Parameters

The final stage examined whether the parameters specific to each priority event were actually being captured. Volume can look normal while underlying field-level capture degrades — this stage is where that would surface.

Story events. The October 29 spike in story volume coincided directly with a spike in missing title and url values. The increased load on the site may have degraded parameter capture during that window. Missing text was persistently high throughout the entire period — roughly 90%+ — but this appears to be a structural characteristic of story events rather than a regression, since story submissions typically don't include body text.

Screenshot: Missing Parameter Capture — Story Event (By Date)

Comment events. Comment parameter capture showed no corresponding degradation during the October spike period. Missing text on comments stayed stable and low throughout the trailing 12 months, which helps isolate the parameter capture issue to the story event pipeline specifically. Notably, descendants, ranking, and deleted were never populated on comment events during this period — consistent with those fields being either deprecated or not applicable to the comment event type.

Screenshot: Missing Parameter Capture — Comment Event (By Date)
Story event parameter capture by date 
SELECT
    FORMAT_DATE('%Y-%m-%d', DATE(timestamp, "America/New_York")) AS event_date,
    COUNT(*) AS events,
    COUNTIF(title IS NULL) AS missing_title,
    COUNTIF(url IS NULL) AS missing_url,
    COUNTIF(text IS NULL) AS missing_text,
    COUNTIF(`by` IS NULL) AS missing_by,
    COUNTIF(score IS NULL) AS missing_score
FROM `bigquery-public-data.hacker_news.full`
WHERE type = 'story'
AND DATE(timestamp, "America/New_York") BETWEEN '2024-04-01' AND '2025-03-31'
GROUP BY 1
ORDER BY 1 ASC;
Comment event parameter capture by date 
SELECT
    FORMAT_DATE('%Y-%m-%d', DATE(timestamp, "America/New_York")) AS event_date,
    COUNT(*) AS events,
    COUNTIF(text IS NULL) AS missing_text,
    COUNTIF(parent IS NULL) AS missing_parent,
    COUNTIF(descendants IS NULL) AS missing_descendants,
    COUNTIF(ranking IS NULL) AS missing_ranking,
    COUNTIF(deleted IS NULL) AS missing_deleted
FROM `bigquery-public-data.hacker_news.full`
WHERE type = 'comment'
AND DATE(timestamp, "America/New_York") BETWEEN '2024-04-01' AND '2025-03-31'
GROUP BY 1
ORDER BY 1 ASC;

Automation Enhancements

Ad-hoc validation after every major release isn't scalable. The right answer is a set of automated alerts on the daily ETL process that would have flagged the October 29 spike proactively.

The alert logic would monitor two signals: variance in daily event volume by type (a threshold like +/- 50% of the rolling daily average would have caught the Oct 29 story spike), and variance in missing parameter rates on key fields like title and url for story events. Alerts would route to a monitoring Slack channel or email, with a daily validation dashboard as a secondary review layer following each data load.

Retrospective

The exploration and structuring phase took the most time — but that investment paid off. Once the framework was clear, execution moved quickly. Spending time on the approach before writing queries meant every query had a defined purpose rather than being exploratory.

The single-table architecture on hacker_news.full is worth a structural critique: storing multiple event types in one table produces a sparse schema with high null density across columns. This is both a query performance issue and a maintenance cost. Separate event tables would be preferable, and the absence of a primary key constraint creates real join risk if this table is ever used in a broader data model.

BigQuery's interface splits across Google Cloud Console and the BigQuery environment, which adds navigation friction during multi-step validation work. Query history is also separated from table exploration, and results can't be saved or managed directly in the tool. For organized ad-hoc work, exporting to Google Sheets mid-analysis was the workaround.
